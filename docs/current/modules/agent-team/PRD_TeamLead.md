## 一、概述

### 1.1 背景与现状

#### 业务场景

- **使用主体**：终端用户；主会话 Agent（创建 Team 后升任 Team Lead）；Team 内独立运行的队员（Teammate）。
- **核心诉求**：对「同时重构多个模块」「多角度排查同一问题」等**持续性、需互相交流**的工作，希望多人并行推进，队员之间可横向对齐任务与进度，Lead 专注拆解、派发、审批与收敛，而不是兼任「大脑 + 邮局」。
- **现有方案及局限**：
  - SubAgent 已实现委派隔离（消息、权限账本、读缓存、token 计数等），但拓扑仍是**星型**：子 Agent 只能与主 Agent 通信。
  - Worktree 已提供独立工作目录与文件系统级并发安全，但子 Agent 之间仍无横向通道。
  - 主 Agent 既要决策又要中转，易成吞吐瓶颈。

#### 现状的核心问题

1. **星型拓扑无横向通道**：队员之间不能直连，所有协作信息必须经 Lead 中转。
2. **缺少长期小组对象**：无法固化名称、负责人、花名册与持久化位置，跨轮指派成本高。
3. **完成后难续派**：队员自然结束后若无法从磁盘恢复上下文，只能重新完整派生。
4. **协调与执行角色混杂**：缺少可选的 Coordinator 模式，Lead 仍可能自己改代码，与「只调度、收敛」目标冲突。
5. **运行环境差异**：需覆盖终端强隔离窗格与同进程轻量执行，且一经选定不得静默回退。

#### 解决方案目标

将 LavenderCode 从星型升级为**网状**：主 Agent 创建长期 **Team** 后升任 **Lead**；每个队员是独立 Agent 实例（独立对话与独立 Worktree）；队员经**共享任务列表**与**邮箱**直连；协作工具仅在 Team 上下文出现；支持暂停/续写；可选 **Coordinator Mode**（双锁）剥夺写文件类工具，保留调度、读类与 Shell（用于 git 合并）；收敛阶段由 Lead 用 Shell 逐个合并队员分支，冲突由推理解决，搞不定则中止合并并上报。

一期只做到「Lead 多人协作 + Plan 审批 + Coordinator 收敛」；跨进程跨机器分布式、队员实时流式通信、复杂任务依赖约束（优先级/deadline）、Windows 平台 iTerm2 适配不在范围。

### 1.2 产品目标

- **G1**：提供长期小组 Team 及其生命周期管理（名称、负责人、成员花名册、配置位置）；单进程内可管理多个 Team（典型同时仅一个活跃）。
- **G2**：主 Agent 可通过「创建小组」能力一键建组：检测执行后端、落盘配置、将 Lead 登记为第一成员；同名自动后缀避免冲突。
- **G3**：扩展「派生成员」能力：指定小组名时走 Team 派生命令路径（独立 Worktree、协作工具、后端分流、花名册登记）。
- **G4**：支持删除小组：仅当无活跃成员（或强制）时清理队员工作区、会话与小组目录，Lead 退出团队。
- **G5**：提供三种执行后端（tmux / iTerm2 / 同进程），按环境优先级一次性检测选定，启动后不静默回退。
- **G6**：队员具备共享任务增删查改（含依赖字段）与点对点消息；主 Agent 与普通 SubAgent 不可见这些工具。
- **G7**：消息支持按成员名、实例标识或全员广播寻址；强隔离后端可唤醒目标窗格。
- **G8–G10**：邮箱并发安全落盘；支持纯文本与多种结构化协议消息；未读消息在队员下一轮工作开始前注入，并批量标已读。
- **G11–G13**：支持定义式派生与可选的 Fork 派生；自然结束后空闲通知 Lead；Lead 再发消息可从磁盘恢复上下文续派。
- **G14**：需 Plan 审批的队员以计划模式起步，经 Lead 结构化批复后再开干。
- **G15**：Coordinator Mode 独立于 Team，配置能力开关 + 环境变量双锁全开才生效；开启后 Lead 工具收窄至调度/读/Shell，并注入四阶段协调纪律提示。
- **G16**：收敛不提供专用合并工具——由 Lead 用 Shell 逐个合并，冲突可解则解，否则中止合并、保留队员工作区并上报用户。
- **G17**：提供 `/team` 系列人工介入命令（列表、详情、删除、终止成员）。
- **G18**：与既有 SubAgent/后台任务/续派等能力协同：未建组时主 Agent 工具面稳定；队员续派复用既有后台任务续派通道。

### 1.3 术语表

| 术语 | 含义 |
| :--- | :--- |
| Team / 小组 | 长期存在的协作单元：名称、负责人、成员花名册、持久化配置位置、默认执行后端 |
| Team Lead / Lead | 创建 Team 后的主会话负责人；负责拆解、派生、审批、合并与用户对接 |
| 队员 / Teammate | Team 内独立 Agent 实例，有独立对话与独立 Worktree |
| 普通 SubAgent | 非 Team 协作形态的子 Agent；不注入小队协作工具 |
| 共享任务列表 | Team 内共享的任务集合，支持增删查改与阻塞依赖字段 |
| 邮箱 | 按收件人落盘的消息收件箱；经名称注册表解析后投递 |
| 名称注册表 | 成员名 ↔ 实例标识的双向查询能力，供消息寻址 |
| 执行后端 | 队员运行环境：`tmux` / `iterm2` / `in-process`（同进程） |
| Coordinator Mode | 可选协调者模式：双锁启用后剥夺 Lead 写文件/编辑文件类工具 |
| Plan 审批 | `planModeRequired` 队员先出计划，Lead 用协议消息批准或驳回 |
| 空闲通知 | 队员自然结束后标记非活跃，并向 Lead 邮箱投递可再指派通知 |
| 续写 / 续派 | 对已停止队员发消息，从会话磁盘恢复对话后继续执行 |
| 收敛 | 全部相关工作完成后，Lead 用 git 合并各队员分支并处理冲突 |
| 双锁 | 配置中的能力开关与环境变量同时打开，Coordinator 才生效 |

## 二、角色与权限

### 2.1 角色定义

| 角色 | 职责描述 | 主要使用方式（业务入口） |
| :--- | :--- | :--- |
| 终端用户 | 提出目标；决定是否启用 Coordinator；用 `/team` 人工介入；接收合并/冲突上报 | 主会话对话、环境变量、Slash 命令 |
| Team Lead | 创建/删除小组、拆解任务、派生成员、发消息、审批 Plan、Shell 合并、终止成员 | 主会话在 TeamCreate 后升任；可选 Coordinator |
| 队员 | 更新共享任务、执行子目标、点对点/广播协作、完成后空闲待机、按消息续写 | Team 派生实例；仅队员可见协作工具 |
| 需 Plan 审批的队员 | 同队员，但必须以计划模式起步，获批后方可按 Lead 当前权限模式执行 | 派生时标记 `planModeRequired` |
| 普通 SubAgent | 既有委派执行，不参与小队任务/邮箱 | 主入口原有 Agent 委派（不带小组名） |

### 2.2 权限矩阵

| 操作 | 用户 | Lead（普通） | Lead（Coordinator） | 队员（同进程） | 队员（窗格后端） | 需 Plan 队员 | 普通 SubAgent |
| :--- | :---: | :---: | :---: | :---: | :---: | :---: | :---: |
| 创建小组 TeamCreate | - | ✓ | ✓ | - | - | - | - |
| 删除小组 TeamDelete | ✓（`/team delete`） | ✓ | ✓ | - | - | - | - |
| 向小组派生成员 | - | ✓ | ✓ | ✗（拦截） | ✗（不可往 Team 加人） | ✗ | ✗ |
| 共享任务 CRUD | - | 条件见下 | ✓（白名单内） | ✓ | ✓ | ✓ | ✗ |
| SendMessage | - | ✓（协调用） | ✓ | ✓ | ✓ | ✓ | ✗ |
| 发送 plan_approval_response | - | ✓（仅 Lead） | ✓ | ✗ | ✗ | ✗ | ✗ |
| 读类工具 | - | ✓ | ✓ | ✓ | ✓ | ✓（获批后按模式） | 按既有 |
| 写文件/编辑文件 | - | ✓ | **剥夺** | ✓ | ✓ | ✓（获批后） | 按既有 |
| Shell / bash（含 git merge） | - | ✓ | ✓ | ✓ | ✓ | ✓（获批后） | 按既有 |
| `/team` 人工介入 | ✓ | - | - | - | - | - | - |

> 说明：协作工具对「未建组主 Agent」不可见；`TeamCreate` / `TeamDelete` 对主 Agent 始终可见。Coordinator 开启后 Lead 允许工具仅限调度、任务、消息、读类与 bash。

### 3.2 系统级主流程（端到端）

1. **[阶段一] 建组**：用户或 Lead 发起创建小组 → 一次性检测执行后端 → 落盘小组配置 → Lead 登记为第一成员。
2. **[阶段二] 拆解与派工**：Lead 将目标拆入共享任务（可含阻塞依赖）→ 按小组名派生成员（独立 Worktree + 协作工具 + 后端启动）→ 花名册与名称注册表更新。
3. **[阶段三] 网状执行**：队员更新任务、经邮箱点对点或广播协作；未读消息在下一轮工作开始前注入；Lead 也可收到队员更新（含空闲时自动续推，避免卡在等待用户输入）。
4. **[阶段四] Plan 审批（条件）**：需审批队员以计划模式起步 → 将计划发给 Lead → Lead 以批准/驳回协议回复 → 批准则切到可执行模式，驳回则按反馈改计划再提。
5. **[阶段五] 空闲与续派**：队员自然结束后标记非活跃并通知 Lead → Lead 再发消息时从会话磁盘恢复上下文继续（同进程）；窗格后端凭邮箱+唤醒续读，窗格已死后报错由 Lead 决定是否重派。
6. **[阶段六] 收敛**：相关任务完成后，Lead 用 Shell 逐个合并队员分支；冲突自行解决或中止合并并上报，**保留**队员工作区。
7. **[终态]**：用户确认合并成果或收到冲突上报；可用 `/team delete`（无活跃成员或强制）清理小组资源。

**分支与异常（业务语言）：**

- 后端检测后不可用/启动失败 → 明确失败，**不**静默改用其他后端。
- 有活跃成员时非强制删除 → 拒绝删除。
- 同进程队员尝试再向小组派生成员 → 拒绝。
- 邮箱锁竞争 → 抖动重试；锁过旧视为过期后可继续。
- Coordinator 仅开一锁 → 不生效；一旦生效，当次进程内不可运行时解锁（须退出重启取消）。

### 3.3 与外部协作的关系（业务级）

| 外部协作方 | 业务上依赖什么 | 本平台对外提供什么（业务结果） | 备注 |
| :--- | :--- | :--- | :--- |
| 本机 Git / Worktree | 队员隔离目录与分支、合并与中止合并 | 合并后的统一成果或冲突上报 | 无专用合并工具，由 Lead Shell 执行 |
| tmux | 横向分屏完整实例、唤醒与终止窗格 | 强隔离队员运行 | 会话外可有 detached 路径；失败不回退同进程 |
| iTerm2 + 配套 CLI | 分屏启动、唤醒、关闭窗格 | 强隔离队员运行（主目标环境含 macOS） | Windows 特殊适配一期不做 |
| 用户主目录小组配置区 | 可写的按小组名分目录 | 配置、任务清单、邮箱与锁文件长期保存 | 路径口径：用户主目录下 LavenderCode 团队目录 |

## 五、非功能需求（NFR）

### 5.1 性能

| 指标 | 目标值 | 备注 |
| :--- | :--- | :--- |
| 邮箱投递可见性 | 窗格唤醒路径下目标在约 0～2 秒内响应轮询；同进程下一轮循环自动读信 | 含锁重试 |
| Lead 收信 | 后台约每秒轮询 Lead 邮箱；空闲时可自动开启一轮处理 | 避免提醒静默堆积 |
| 并发写同一邮箱 | 多路并发写入最终不丢不截断 | 文件锁 + 原子替换 |
| 容量 | 单机、单活跃 LavenderCode 实例操作活跃 Team；不承诺跨机器 | 一期边界 |

### 5.2 可用性

| 检查项 | 目标 / 要求 |
| :--- | :--- |
| 核心功能可用性 | 建组、派发、任务、消息、空闲续派、删除、`/team` 在目标环境可用 |
| 单点故障影响 | 单队员异常不破坏其他成员邮箱与花名册；合并失败须可中止并上报 |
| 降级方案 | **禁止**执行后端静默回退；邮箱锁冲突重试，过期锁可回收 |

### 5.3 安全

| 检查项 | 要求 |
| :--- | :--- |
| 权限控制 | 协作工具仅 Team 队员上下文；`plan_approval_response` 仅 Lead；Coordinator 剥夺写/编辑文件；队员一律免交互确认（无 TUI 接审批弹窗），安全靠工具白名单 + Worktree + Plan |
| 数据安全 | 小组数据按用户主目录分团队隔离；跨 Team 不可寻址 |
| 审计追溯 | 建议记录建组/派发/审批/合并或中止/Coordinator 生效（见附录） |
| 临时目录 | 文件类工具允许写入系统临时目录白名单（避免正常中转被沙箱误杀）；Shell 仍按执行类权限，不受该文件沙箱约束 |

### 5.4 兼容性

| 检查项 | 要求 |
| :--- | :--- |
| 存量行为 | 未 TeamCreate 时主 Agent 工具列表保持稳定；`TeamCreate`/`TeamDelete` 始终可见；不带小组名的 Agent 委派维持原行为 |
| 既有续派 | 普通后台任务续派路径保留；Team 队员续派复用同一底层任务管理能力 |
| 客户端环境 | 一期以 macOS / Linux 为主；iTerm2 后端主目标非 Windows；tmux 在 WSL 不保证 |

### 5.5 可维护性

| 检查项 | 要求 |
| :--- | :--- |
| 日志 / 输出 | 错误消息与 TUI 输出中文友好；窗格内队员以只读日志流呈现 |
| 配置管理 | Coordinator：`features.coordinatorMode`（或等价能力开关）+ 环境变量双锁；进程内不可运行时解锁 |
| 状态一致性 | 跨进程更新花名册时，写盘前先重载磁盘成员再改写，避免丢更新 |

### 5.6 可测试性

| 检查项 | 说明 |
| :--- | :--- |

## 九、附录

### 9.1 业务异常与提示口径（可选）

| 业务异常类别 | 典型触发场景 | 面向用户的提示原则 |
| :--- | :--- | :--- |
| 执行后端不可用/启动失败 | 检测或 spawn 失败 | 说明原因；明确未改用其他后端 |
| 小组不存在 | 派发或删除错误名 | 提示名称无效或未创建 |
| 仍有活跃成员 | 非强制删除 | 提示先等空闲或使用强制删除 |
| 同进程禁止再派生 | 队员尝试往 Team 加人 | 明确仅 Lead 可向小组派生成员 |
| 邮箱锁繁忙 | 高并发写入 | 可重试；持续失败提示稍后再发 |
| 窗格已死无法续写 | 对已消失窗格发消息 | 报错，由 Lead 决定是否重新派生 |
| Plan 未获批 | 需审批队员 | 须等待 Lead 批准/驳回协议消息 |
| Coordinator 未双开 | 只开配置或只开环境变量 | 说明需两把锁同时打开 |
| 合并冲突不可解 | 收敛阶段 | 已中止合并；保留队员工作区；报告冲突文件与路径 |
